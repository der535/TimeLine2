# Как комментировать скрипты?

Самое главное не пишим воду.
Например: // Сериализованные поля - хорошая практика для настройки через Inspector
Этот комментарий ничего не даёт тому кто читает этот код.

Комментировать поля вообще не надо, т.к описание должно быть внутри самого класса, если это поле простой тип данных типа строки или int то всё равно не надо его коментировать так как название этого поля само за себя говорит что оно делает.

Плохой пример:
```cs
[SerializeField] private PlayerHitController playerHitController;
[SerializeField] private PlayerInvulnerable playerInvulnerable;
        // Сериализованные поля - хорошая практика для настройки через Inspector
SerializeField] private PlayerHitController playerHitController; // Пока не используется в коде
[SerializeField] private PlayerInvulnerable playerInvulnerable; // Важно для проверки неуязвимости
```

Не нужно комментировать конструкторы
Не нужно комментировать namespace

Ещё плохой пример
```cs
        // Обработка физических столкновений (Collider + Rigidbody)
        private void OnCollisionEnter2D(Collision2D collision)
        {
            if (collision.gameObject.CompareTag(TagsStorage.IsDamageable) && !playerInvulnerable.IsInvulnerable)
            {
                // Использование EventBus - хорошая архитектурная практика
                _gameEventBus.Raise(new PlayerTakeDamageEvent());
            }
        }
```
Тут лучше бы добавить комментарий описывающий условие ведь он тут замудрённый

Хороший пример
```cs
        private void OnCollisionEnter2D(Collision2D collision)
        {
        // Если у препятствия тег IsDamageable и игрок уязвим то наносим урон
            if (collision.gameObject.CompareTag(TagsStorage.IsDamageable) && !playerInvulnerable.IsInvulnerable)
            {
             
                _gameEventBus.Raise(new PlayerTakeDamageEvent());
            }
        }

```

Если есть замечания по коду, например нет отписки от событий или неиспользуемое поле, пиши прямо в коде 
```cs
//todo и дальше что надо
```

# Название коммитов

**НЕ ТРОГАЙ ВЕТКУ MAIN**

### Правила названий коммитов в ветке MAIN
В названии коммита номер версии допустим 2.1 с каждым коммитом + 1 десятая. Тоесть 1.0 -> 1.1 -> 1.2 -> 1.3 и .тд 

### Правила названий веток
Берём в название базовую версию и добавляем в конец точку и порядковый номер коммита в этой ветке. Например 1.0 база и это первый коммит в новой ветке значит 1 получается 1.0.1 если второй коммит 1.0.2 если допустим 12 коммит то 1.0.12 дальше пишем в описание через минус перечисление всех изменений например:
```
- Оптимизированно выделение трек обжектов
- Исправленны баги с курсором
- Исправленны баги с ключевыми кадрами
```
